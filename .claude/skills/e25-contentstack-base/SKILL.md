# e25-contentstack-base

Invoked via `/e25-contentstack-base` (or matching natural-language requests). Runs the full scaffold-and-seed flow end-to-end.

Three-gate flow:

1. **Gate 1 — Ask for the project name.** Required input. Either a kebab-case name (e.g. `acme-poc`, creates a new subdirectory under CWD) or the literal `./` (installs in place into the directory that contains `.claude/`).
2. **Gate 2 — Always run `npx create-next-app@latest`.** Never skip this step. Never substitute a cached/pinned Next.js scaffold. The point of this skill is that every new project starts on the latest Next.js + React + Tailwind versions.
3. **Gate 3 — Collect Contentstack credentials and write `.env.local`.** After the overlay is copied in, prompt the user for `CONTENTSTACK_API_KEY`, `CONTENTSTACK_DELIVERY_TOKEN`, `CONTENTSTACK_ENVIRONMENT`, the region, `CONTENTSTACK_PREVIEW_TOKEN`, and `CONTENTSTACK_MANAGEMENT_TOKEN`, then write `.env.local` in one shot.

After the gates, the skill offers to seed the Contentstack stack (creates `seo` + `flexible_page` content types and a sample home page entry) using the management token. Then it runs a validation step that confirms (a) the Contentstack credentials actually work and (b) the scaffolded codebase compiles cleanly. Only then prints "ready" to the user.

## What the overlay adds

- **Contentstack GraphQL client** at `lib/contentstack/client.ts` reading from `CONTENTSTACK_*` env vars (delivery token rides in the `access_token` header; preview swaps host + adds `preview_token`)
- **FlexiblePage + SEO only** at `lib/contentstack/graphql/{queries/flexiblePage.ts,fragments/seo.ts}` — section fragments are added per project and composed into the page query from the registry
- **Catch-all route** at `app/[[...slug]]/page.tsx` that fetches a `flexible_page` by url and hands its sections to `SectionsRenderer`
- **Section registry scaffold** at `lib/sections/` — empty registry, plus the `SectionDefinition` interface, `SectionsRenderer`, and base types
- **Locale system** at `lib/i18n/locale.ts` — URL slug ↔ Contentstack locale code mapping (English + Spanish by default)
- **`components/ARCHITECTURE.md`** — explains the data flow, the switch-case dispatch model, variant routing inside section components, the connection/union mechanics, and the six-step recipe for adding a new section
- **Tailwind CSS variables** for brand tokens in `app/globals.css` (`--brand-primary`, `--brand-primary-hover`, `--text-default`, `--text-muted`, `--bg-muted`)
- **`.env.example`**, **`README.md`**, **`AGENTS.md`** (which points all future agents at `components/ARCHITECTURE.md`)

After the overlay, Gate 3 prompts the user for the credentials and writes `.env.local` directly with the answers. The `CONTENTSTACK_MANAGEMENT_TOKEN` is write-scope; it's persisted in `.env.local` for re-runs of the seed script, but the Next.js runtime never references it.

The overlay also ships **`scripts/seed-contentstack.mjs`** — a re-runnable Node script that uses a management token to create the `seo` and `flexible_page` content types and publish a sample home page entry at url `/`. Step 6 invokes it.

The overlay overwrites the default `app/layout.tsx`, `app/globals.css`, and `README.md` generated by `create-next-app`, and removes `app/page.tsx` (the catch-all route replaces it).

## Contentstack vs. a reference-hydrating CMS — note for the orchestrator

Contentstack resolves references **inline** in one GraphQL round-trip (via `<field>Connection { edges { node { __typename ... } } }` unions), so there is no per-section second fetch. The section registry contributes a **GraphQL fragment + transform + render** per section; `lib/contentstack/pages.ts` composes the page query from those fragments. While the registry is empty, the page query omits `sectionsConnection` entirely — that is why the seeded `flexible_page` content type ships **without** a `sections` field (a Contentstack reference field must list at least one content type in `reference_to`). The user adds the `sections` reference field when they build their first section. None of this changes the steps below; it only matters if you debug the templates.

## Invocation

When triggered, do exactly the following in order. **Use absolute paths everywhere — never `cd`.** Bash calls share a working directory, so a `cd` in one step bleeds into the next. To avoid that class of bug, capture two absolute paths up front and substitute them into every command below:

- **`${SKILL_DIR}`** — the directory containing this `SKILL.md` file. You know this path because the skill runtime told you where it loaded the skill from; use that. The template lives at `${SKILL_DIR}/template/`.
- **`${CLAUDE_PARENT}`** — the absolute path to the directory containing the `.claude/` folder. Compute it once from `${SKILL_DIR}` by going up three levels (`skill-name → skills → .claude → its parent`). Used only when the user supplies the in-place sentinel as the project name.
- **`${PROJECT_DIR}`** — the absolute path to the new project directory. Set it once immediately after Gate 1:
  - If the user gave the literal **`./`** (or `.`) as the project name → `${PROJECT_DIR}` = `${CLAUDE_PARENT}`. This is an "in-place" install — the Next.js scaffold lands directly in the directory where `.claude/` already lives, no new subdirectory.
  - Otherwise → `${PROJECT_DIR}` = `$(pwd)/<project-name>` as captured at the start.

From that point on, every command refers to `${PROJECT_DIR}` (or files under it) by absolute path. If the user's CWD changes between steps for any reason, the absolute paths still work.

### Step 1 — Gate 1: project name

Use `AskUserQuestion` to ask the user for the **project name**. Two accepted shapes:

1. **A kebab-case name** (e.g. `acme-poc`) — creates a new subdirectory under the current working directory and uses the name as the `package.json` `name`.
2. **The literal `./` (or `.`)** — installs the Next.js scaffold **in place**, directly into `${CLAUDE_PARENT}` (the directory that already contains the `.claude/` folder). No new subdirectory is created. Use this when the user already has a project directory set up (with `.claude/` in it) and wants the Next.js project to live at its root.

Do not proceed without a value. Set `${PROJECT_DIR}` according to the rule in the "Path conventions" paragraph above before moving to Step 2.

### Step 2 — Gate 2: run create-next-app

The arg passed to `create-next-app` depends on which shape the user chose in Gate 1:

- **Named subdirectory**: pass the project name verbatim. The new dir is created under the current CWD.

  ```bash
  npx create-next-app@latest <project-name> \
    --ts --tailwind --eslint --app --no-src-dir \
    --import-alias "@/*" --use-npm --yes
  ```

- **In-place (`./` or `.`)**: pass the absolute path of `${CLAUDE_PARENT}` so create-next-app installs into that exact directory regardless of CWD.

  ```bash
  npx create-next-app@latest ${CLAUDE_PARENT} \
    --ts --tailwind --eslint --app --no-src-dir \
    --import-alias "@/*" --use-npm --yes
  ```

  Heads-up for the in-place case: `create-next-app` may refuse if `${CLAUDE_PARENT}` contains conflicting files (anything other than `.claude/`, `.git/`, and similar dotfiles). If it errors with "directory not empty" or similar, surface the message and stop — ask the user to either move the conflicting files out of the way or pick a new subdirectory name instead. Do not delete files to make space.

After this completes, the project exists at `${PROJECT_DIR}`. The `--yes` flag accepts defaults for any prompts the latest version may introduce. For any other failure, surface the error and stop — do not retry destructively.

### Step 3 — Overlay the Contentstack integration

Copy the overlay using absolute paths (no `cd`):

```bash
cp -R ${SKILL_DIR}/template/. ${PROJECT_DIR}/
```

The trailing `/.` is intentional so dotfiles (`.env.example`) are included. This overwrites `app/layout.tsx`, `app/globals.css`, and `README.md`; everything else under `lib/`, `components/`, `app/[[...slug]]/`, etc. is net-new.

### Step 4 — Remove `app/page.tsx`

The catch-all route at `app/[[...slug]]/page.tsx` handles the root. Delete the conflicting `app/page.tsx` that `create-next-app` generated:

```bash
rm ${PROJECT_DIR}/app/page.tsx
```

### Step 5 — Gate 3: collect Contentstack credentials and write `.env.local`

Ask the user for the values below in a **single plain-text message** (do not use `AskUserQuestion` — its choice-list format is wrong for free-text credentials). Tell them they can reply "skip" for any optional value they want to leave blank.

1. **CONTENTSTACK_API_KEY** — required. Stack API key from Settings → Tokens (a.k.a. the stack's API key).
2. **CONTENTSTACK_DELIVERY_TOKEN** — required. Delivery token for the environment you'll read from. Delivery tokens are **environment-specific**, so it must match the environment below.
3. **CONTENTSTACK_ENVIRONMENT** — required. Publishing environment name (e.g. `development`, `production`). Contentstack has no default environment.
4. **Region** — optional, defaults to **US (North America)**. If the user names a different region, substitute the host prefixes accordingly (see the region table below). If they don't mention it, use the US defaults.
5. **CONTENTSTACK_PREVIEW_TOKEN** — optional. Preview token for Live Preview (draft content). Safe to skip if not using preview mode.
6. **CONTENTSTACK_MANAGEMENT_TOKEN** — optional but recommended. Management token with **write access** to the stack. Used by `scripts/seed-contentstack.mjs` to create content types and seed entries. Skip if you'll manage content types manually. Warn the user explicitly that this token grants write access to their stack — they should keep `.env.local` out of any commit, deploy, or shared screenshot. Generate at Contentstack → Settings → Tokens → Management Tokens.

Region host table (used to fill the `*_HOST` values):

| Region   | GraphQL host                         | Preview GraphQL host                          | API host (seed)                  |
| -------- | ------------------------------------ | --------------------------------------------- | -------------------------------- |
| US (default) | `graphql.contentstack.com`       | `graphql-preview.contentstack.com`            | `api.contentstack.io`            |
| EU       | `eu-graphql.contentstack.com`        | `eu-graphql-preview.contentstack.com`         | `eu-api.contentstack.com`        |
| Azure NA | `azure-na-graphql.contentstack.com`  | `azure-na-graphql-preview.contentstack.com`   | `azure-na-api.contentstack.com`  |
| Azure EU | `azure-eu-graphql.contentstack.com`  | `azure-eu-graphql-preview.contentstack.com`   | `azure-eu-api.contentstack.com`  |
| GCP NA   | `gcp-na-graphql.contentstack.com`    | `gcp-na-graphql-preview.contentstack.com`     | `gcp-na-api.contentstack.com`    |

Unlike some CMS tokens, Contentstack management tokens have **no fixed prefix**, so do not validate a prefix — just confirm the value is non-empty.

Then construct `${PROJECT_DIR}/.env.local` directly with `Write` (do not `cp .env.example .env.local` and then patch line-by-line — write the final file in one shot), filling the hosts from the region table:

```
CONTENTSTACK_API_KEY=<value>
CONTENTSTACK_DELIVERY_TOKEN=<value>
CONTENTSTACK_ENVIRONMENT=<value>

CONTENTSTACK_BRANCH=
CONTENTSTACK_PREVIEW_TOKEN=<value-or-blank>
CONTENTSTACK_GRAPHQL_HOST=<region graphql host>
CONTENTSTACK_PREVIEW_GRAPHQL_HOST=<region preview graphql host>

CONTENTSTACK_API_HOST=<region api host>
CONTENTSTACK_MANAGEMENT_TOKEN=<value-or-blank>
```

If the user skipped any of the three required values (`CONTENTSTACK_API_KEY`, `CONTENTSTACK_DELIVERY_TOKEN`, `CONTENTSTACK_ENVIRONMENT`), leave those lines with empty values and remember to remind them in Step 8 that those three are required for the app to run.

### Step 6 — Seed Contentstack (create content types + sample page)

The seed script needs the `CONTENTSTACK_MANAGEMENT_TOKEN` collected in Gate 3 (plus `CONTENTSTACK_API_KEY` and `CONTENTSTACK_ENVIRONMENT`).

1. If Gate 3 did **not** capture a `CONTENTSTACK_MANAGEMENT_TOKEN` (user skipped it), tell the user the skill can create the `seo` + `flexible_page` content types and a sample `/` entry if they paste a management token now. If they decline, skip this step and remember to flag in Step 8 that `/` will 404 until they manually create the content types and an entry.

   If they agree, ask for the token and write it into `${PROJECT_DIR}/.env.local` by editing the existing `CONTENTSTACK_MANAGEMENT_TOKEN=` line.

2. Run the seed script with an absolute path (no `cd`):

   ```bash
   node ${PROJECT_DIR}/scripts/seed-contentstack.mjs
   ```

   The script auto-loads `${PROJECT_DIR}/.env.local` (resolved from its own location, independent of CWD) and reads `CONTENTSTACK_API_KEY`, `CONTENTSTACK_ENVIRONMENT`, `CONTENTSTACK_API_HOST`, and `CONTENTSTACK_MANAGEMENT_TOKEN` from it. It is idempotent — re-running updates existing content types and entries (matched by url / title) instead of duplicating them.

3. If the script exits non-zero, surface its stderr to the user. Common failures:
   - 401 / 403 → management token invalid or lacks write scope
   - 412 → `api_key` / token mismatch (wrong stack)
   - 422 → field/validation conflict with an existing content type of the same uid, or the publish environment name doesn't exist on the stack

   Do **not** retry destructively. Stop and report.

### Step 7 — Validate setup

Run two checks and report both results to the user. Do not skip this step. Track the pass/fail of each so Step 8 can branch accordingly.

**Check A — Contentstack connectivity (end-to-end).** Skip only if the user did not provide all three of `CONTENTSTACK_API_KEY`, `CONTENTSTACK_DELIVERY_TOKEN`, and `CONTENTSTACK_ENVIRONMENT` in Gate 3. If skipped, mark Check A as `not-run`.

If Step 6 seeded the stack, query for the sample `/` entry — this verifies credentials, the `flexible_page` content type, and the seed entry in one round-trip. Read the env values out of `.env.local` yourself rather than `source`ing the file (values may contain shell-special characters):

```bash
API_KEY=$(grep -E '^CONTENTSTACK_API_KEY=' ${PROJECT_DIR}/.env.local | cut -d= -f2-)
DELIVERY_TOKEN=$(grep -E '^CONTENTSTACK_DELIVERY_TOKEN=' ${PROJECT_DIR}/.env.local | cut -d= -f2-)
ENVIRONMENT=$(grep -E '^CONTENTSTACK_ENVIRONMENT=' ${PROJECT_DIR}/.env.local | cut -d= -f2-)
GQL_HOST=$(grep -E '^CONTENTSTACK_GRAPHQL_HOST=' ${PROJECT_DIR}/.env.local | cut -d= -f2-)
GQL_HOST=${GQL_HOST:-graphql.contentstack.com}

status=$(curl -s -o /tmp/contentstack-probe.json -w "%{http_code}" \
  -X POST "https://${GQL_HOST}/stacks/${API_KEY}?environment=${ENVIRONMENT}" \
  -H "access_token: ${DELIVERY_TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"query":"{ all_flexible_page(where: { url: \"/\" }, limit: 1) { items { system { uid } title url } } }"}')
echo "HTTP $status"
cat /tmp/contentstack-probe.json
```

Classify the outcome of Check A:

- **`pass`** — 200 + non-empty `items`. Connectivity OK, credentials valid, sample entry present.
- **`pass-empty`** — 200 + empty `items`. Connectivity OK; sample entry missing (Step 6 was skipped or the entry was deleted/unpublished to this environment).
- **`schema-missing`** — 200 + a GraphQL error about `all_flexible_page` not existing. Content type wasn't created (Step 6 was skipped or failed).
- **`bad-credentials`** — 401 / 403 / 412. The `access_token` (delivery token) or `api_key` is wrong, or the delivery token doesn't match this environment.
- **`bad-stack`** — 404 / 422 on the `/stacks/...` path. The `CONTENTSTACK_API_KEY` or `CONTENTSTACK_ENVIRONMENT` is invalid.
- **`other`** — anything else. Relay the response body verbatim.

**Check B — codebase compiles cleanly.** Always run.

```bash
npm --prefix ${PROJECT_DIR} run build
```

`next build` runs the TypeScript type-checker and ESLint. Any failure means the overlay didn't sit cleanly on top of the `create-next-app` scaffold (e.g. a Next.js API changed, the `@/*` alias didn't resolve, or a dependency is missing).

Classify the outcome of Check B:

- **`pass`** — exit 0.
- **`fail`** — non-zero. Surface the first error block. Do **not** attempt to auto-patch the scaffold; this is a signal that the skill template needs to be updated for the current `create-next-app@latest` output.

### Step 8 — Print next steps (do NOT run them)

Branch on the Check A / Check B results recorded in Step 7. Use the table below to choose the message. Print only the section that matches.

**Both green (`pass` + `pass`):**

```
✓ Validated against Contentstack. ✓ Build clean.

Ready:
  cd <project-name>
  npm run dev
```

Then point the user at `components/ARCHITECTURE.md` for the six-step recipe to add their first section.

**Check A `pass-empty` or `schema-missing`, Check B `pass`:**

The app will run but `/` will return 404 because the home page entry doesn't exist (or the content types weren't created / weren't published to this environment). Tell the user one of:
- Re-run Step 6 (`node ${PROJECT_DIR}/scripts/seed-contentstack.mjs` with a management token), or
- Create the `flexible_page` and `seo` content types and a `flexible_page` entry with url `/` manually in Contentstack, then publish it to `CONTENTSTACK_ENVIRONMENT`.

Then `npm run dev` will serve `/`.

**Check A `bad-credentials` or `bad-stack`:**

Do **not** tell the user to run `npm run dev` — it will throw at request time. Tell them the specific thing to fix:
- `bad-credentials` → confirm `CONTENTSTACK_DELIVERY_TOKEN` is the delivery token **for this environment** and `CONTENTSTACK_API_KEY` is correct, fix in `${PROJECT_DIR}/.env.local`, re-run validation.
- `bad-stack` → fix `CONTENTSTACK_API_KEY` / `CONTENTSTACK_ENVIRONMENT` in `${PROJECT_DIR}/.env.local`, re-run validation.

**Check A `not-run` (user skipped credentials in Gate 3):**

Tell the user they must fill in `CONTENTSTACK_API_KEY`, `CONTENTSTACK_DELIVERY_TOKEN`, and `CONTENTSTACK_ENVIRONMENT` in `${PROJECT_DIR}/.env.local` before `npm run dev` will serve any page.

**Check B `fail` (regardless of Check A):**

Do **not** tell the user to run `npm run dev`. The build is broken; surface the error block and stop. The template needs a fix before this scaffold is usable.

## What this skill does NOT do

- Does not pin a Next.js version — relies on whatever `create-next-app@latest` ships.
- Does not ship example sections — only FlexiblePage + SEO. Sections are added per project per the architecture doc.
- Does not configure **section** content models. Step 6 creates `flexible_page` and `seo` in the user's stack when they provide a management token; any section types they build later (Hero, CardGroups, etc.) — and the `sections` reference field on `flexible_page` — are their responsibility.
- Does not wire full Live Preview (real-time hash updates) — the client supports preview host + `preview_token`, but the Live Preview SDK and edit tags are added per project.
- Does not pull from any client-specific repo.
