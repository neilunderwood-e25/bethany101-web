# e25-contentful-base

Invoked via `/e25-contentful-base` (or matching natural-language requests). Runs the full scaffold-and-seed flow end-to-end.

Three-gate flow:

1. **Gate 1 — Ask for the project name.** Required input. Either a kebab-case name (e.g. `acme-poc`, creates a new subdirectory under CWD) or the literal `./` (installs in place into the directory that contains `.claude/`).
2. **Gate 2 — Always run `npx create-next-app@latest`.** Never skip this step. Never substitute a cached/pinned Next.js scaffold. The point of this skill is that every new project starts on the latest Next.js + React + Tailwind versions.
3. **Gate 3 — Collect Contentful credentials and write `.env.local`.** After the overlay is copied in, prompt the user for `CONTENTFUL_SPACE_ID`, `CONTENTFUL_DELIVERY_TOKEN`, `CONTENTFUL_PREVIEW_TOKEN`, and `CONTENTFUL_MANAGEMENT_TOKEN`, then write `.env.local` in one shot.

After the gates, the skill offers to seed the Contentful space (creates `seo` + `flexiblePage` content types and a sample home page entry) using the CMA token. Then it runs a validation step that confirms (a) the Contentful credentials actually work and (b) the scaffolded codebase compiles cleanly. Only then prints "ready" to the user.

## What the overlay adds

- **Contentful GraphQL client** at `lib/contentful/client.ts` reading from `CONTENTFUL_*` env vars
- **FlexiblePage + SEO queries only** at `lib/contentful/graphql/{queries/flexiblePage.ts,fragments/seo.ts}` — section queries are added per project
- **Catch-all route** at `app/[[...slug]]/page.tsx` that fetches a `FlexiblePage` by slug and hands its sections to `SectionsRenderer`
- **Section registry scaffold** at `lib/sections/` — empty registry, plus the `SectionDefinition` interface, `SectionsRenderer`, and base types
- **Locale system** at `lib/i18n/locale.ts` — URL slug ↔ Contentful locale code mapping (English + Spanish by default)
- **`components/ARCHITECTURE.md`** — explains the data flow, the switch-case dispatch model, variant routing inside section components, and the six-step recipe for adding a new section
- **Tailwind CSS variables** for brand tokens in `app/globals.css` (`--brand-primary`, `--brand-primary-hover`, `--text-default`, `--text-muted`, `--bg-muted`)
- **`.env.example`**, **`README.md`**, **`AGENTS.md`** (which points all future agents at `components/ARCHITECTURE.md`)

After the overlay, Gate 3 prompts the user for `CONTENTFUL_SPACE_ID`, `CONTENTFUL_DELIVERY_TOKEN`, `CONTENTFUL_PREVIEW_TOKEN`, and `CONTENTFUL_MANAGEMENT_TOKEN` and writes `.env.local` directly with the answers. The `CONTENTFUL_MANAGEMENT_TOKEN` is write-scope; it's persisted in `.env.local` for re-runs of the seed script, but the Next.js runtime never references it.

The overlay also ships **`scripts/seed-contentful.mjs`** — a re-runnable Node script that uses a Content Management API (CMA) token to create the `seo` and `flexiblePage` content types and publish a sample home page entry at slug `/`. Step 6 invokes it.

The overlay overwrites the default `app/layout.tsx`, `app/globals.css`, and `README.md` generated by `create-next-app`, and removes `app/page.tsx` (the catch-all route replaces it).

## Invocation

When triggered, do exactly the following in order. **Use absolute paths everywhere — never `cd`.** Bash calls share a working directory, so a `cd` in one step bleeds into the next. To avoid that class of bug, capture two absolute paths up front and substitute them into every command below:

- **`${SKILL_DIR}`** — the directory containing this `SKILL.md` file. You know this path because the skill runtime told you where it loaded the skill from; use that. The template lives at `${SKILL_DIR}/template/`.
- **`${CLAUDE_PARENT}`** — the absolute path to the directory containing the `.claude/` folder. Compute it once from `${SKILL_DIR}` by going up three levels (`skill-name → skills → .claude → its parent`). Used only when the user supplies the in-place sentinel as the project name.
- **`${PROJECT_DIR}`** — the absolute path to the new project directory. Set it once immediately after Gate 1:
  - If the user gave the literal **`./`** (or `.`) as the project name → `${PROJECT_DIR}` = `${CLAUDE_PARENT}`. This is an "in-place" install — the Next.js scaffold lands directly in the directory where `.claude/` already lives, no new subdirectory.
  - Otherwise → `${PROJECT_DIR}` = `$(pwd)/<project-name>` as captured at the start.

From that point on, every command refers to `${PROJECT_DIR}` (or files under it) by absolute path. If the user's CWD changes between steps for any reason, the absolute paths still work.

### Step 1 — Gate 1: project name

Use `AskUserQuestion` to ask the user for the **project name**. Two accepted shapes:

1. **A kebab-case name** (e.g. `acme-poc`) — creates a new subdirectory under the current working directory and uses the name as the `package.json` `name`.
2. **The literal `./` (or `.`)** — installs the Next.js scaffold **in place**, directly into `${CLAUDE_PARENT}` (the directory that already contains the `.claude/` folder). No new subdirectory is created. Use this when the user already has a project directory set up (with `.claude/` in it) and wants the Next.js project to live at its root.

Do not proceed without a value. Set `${PROJECT_DIR}` according to the rule in the "Path conventions" paragraph above before moving to Step 2.

### Step 2 — Gate 2: run create-next-app

The arg passed to `create-next-app` depends on which shape the user chose in Gate 1:

- **Named subdirectory**: pass the project name verbatim. The new dir is created under the current CWD.

  ```bash
  npx create-next-app@latest <project-name> \
    --ts --tailwind --eslint --app --no-src-dir \
    --import-alias "@/*" --use-npm --yes
  ```

- **In-place (`./` or `.`)**: pass the absolute path of `${CLAUDE_PARENT}` so create-next-app installs into that exact directory regardless of CWD.

  ```bash
  npx create-next-app@latest ${CLAUDE_PARENT} \
    --ts --tailwind --eslint --app --no-src-dir \
    --import-alias "@/*" --use-npm --yes
  ```

  Heads-up for the in-place case: `create-next-app` may refuse if `${CLAUDE_PARENT}` contains conflicting files (anything other than `.claude/`, `.git/`, and similar dotfiles). If it errors with "directory not empty" or similar, surface the message and stop — ask the user to either move the conflicting files out of the way or pick a new subdirectory name instead. Do not delete files to make space.

After this completes, the project exists at `${PROJECT_DIR}`. The `--yes` flag accepts defaults for any prompts the latest version may introduce. For any other failure, surface the error and stop — do not retry destructively.

### Step 3 — Overlay the Contentful integration

Copy the overlay using absolute paths (no `cd`):

```bash
cp -R ${SKILL_DIR}/template/. ${PROJECT_DIR}/
```

The trailing `/.` is intentional so dotfiles (`.env.example`) are included. This overwrites `app/layout.tsx`, `app/globals.css`, and `README.md`; everything else under `lib/`, `components/`, `app/[[...slug]]/`, etc. is net-new.

### Step 4 — Remove `app/page.tsx`

The catch-all route at `app/[[...slug]]/page.tsx` handles the root. Delete the conflicting `app/page.tsx` that `create-next-app` generated:

```bash
rm ${PROJECT_DIR}/app/page.tsx
```

### Step 5 — Gate 3: collect Contentful credentials and write `.env.local`

Ask the user for the four values below in a **single plain-text message** (do not use `AskUserQuestion` — its choice-list format is wrong for free-text credentials). Tell them they can reply "skip" for any value they want to leave blank.

1. **CONTENTFUL_SPACE_ID** — required. Space ID from the Contentful dashboard.
2. **CONTENTFUL_DELIVERY_TOKEN** — required. Content Delivery API token (read-only, published content).
3. **CONTENTFUL_PREVIEW_TOKEN** — optional. Content Preview API token (draft content). Safe to skip if not using preview mode.
4. **CONTENTFUL_MANAGEMENT_TOKEN** — optional but recommended. CMA token with **write access** to the space. Used by `scripts/seed-contentful.mjs` to create content types and seed entries. Starts with `CFPAT-`. Skip if you'll manage Contentful content types manually. Warn the user explicitly that this token grants full write access to their space — they should keep `.env.local` out of any commit, deploy, or shared screenshot. Generate at Contentful → Settings → API keys → Content management tokens.

If the user provides a `CONTENTFUL_MANAGEMENT_TOKEN`, validate it starts with `CFPAT-`. If it doesn't, point that out and ask them to paste it again (they probably copied the wrong value — Delivery and Preview tokens have different prefixes).

Then construct `${PROJECT_DIR}/.env.local` directly with `Write` (do not `cp .env.example .env.local` and then patch line-by-line — write the final file in one shot):

```
CONTENTFUL_SPACE_ID=<value-or-blank>
CONTENTFUL_DELIVERY_TOKEN=<value-or-blank>

CONTENTFUL_ENVIRONMENT=master
CONTENTFUL_PREVIEW_TOKEN=<value-or-blank>
CONTENTFUL_GRAPHQL_HOST=graphql.contentful.com
CONTENTFUL_PREVIEW_GRAPHQL_HOST=preview.contentful.com

CONTENTFUL_MANAGEMENT_TOKEN=<value-or-blank>
```

If the user skipped `CONTENTFUL_SPACE_ID` or `CONTENTFUL_DELIVERY_TOKEN`, leave those lines with empty values and remember to remind them in Step 8 that those two are required for the app to run.

### Step 6 — Seed Contentful (create content types + sample page)

The seed script needs the `CONTENTFUL_MANAGEMENT_TOKEN` collected in Gate 3.

1. If Gate 3 did **not** capture a `CONTENTFUL_MANAGEMENT_TOKEN` (user skipped it), tell the user the skill can create the `seo` + `flexiblePage` content types and a sample `/` entry if they paste a CMA token now. If they decline, skip this step and remember to flag in Step 8 that `/` will 404 until they manually create the content types and an entry.

   If they agree, ask for the token, validate the `CFPAT-` prefix, and append it to `${PROJECT_DIR}/.env.local` by editing the existing `CONTENTFUL_MANAGEMENT_TOKEN=` line.

2. Run the seed script with an absolute path (no `cd`):

   ```bash
   node ${PROJECT_DIR}/scripts/seed-contentful.mjs
   ```

   The script auto-loads `${PROJECT_DIR}/.env.local` (resolved from its own location, independent of CWD) and reads `CONTENTFUL_SPACE_ID`, `CONTENTFUL_ENVIRONMENT`, and `CONTENTFUL_MANAGEMENT_TOKEN` from it. It is idempotent — re-running updates existing content types and entries instead of duplicating them.

3. If the script exits non-zero, surface its stderr to the user. Common failures:
   - 401 → CMA token invalid or expired
   - 403 → token lacks write scope on this space/environment
   - 422 → field/validation conflict with an existing content type of the same id (have the user inspect the space)

   Do **not** retry destructively. Stop and report.

### Step 7 — Validate setup

Run two checks and report both results to the user. Do not skip this step. Track the pass/fail of each so Step 8 can branch accordingly.

**Check A — Contentful connectivity (end-to-end).** Skip only if the user did not provide both `CONTENTFUL_SPACE_ID` and `CONTENTFUL_DELIVERY_TOKEN` in Gate 3. If skipped, mark Check A as `not-run`.

If Step 6 seeded the space, query for the sample `/` entry — this verifies credentials, the FlexiblePage content type, and the seed entry in one round-trip. Read the env values out of `.env.local` yourself rather than `source`ing the file (values may contain shell-special characters):

```bash
SPACE_ID=$(grep -E '^CONTENTFUL_SPACE_ID=' ${PROJECT_DIR}/.env.local | cut -d= -f2-)
DELIVERY_TOKEN=$(grep -E '^CONTENTFUL_DELIVERY_TOKEN=' ${PROJECT_DIR}/.env.local | cut -d= -f2-)
ENVIRONMENT=$(grep -E '^CONTENTFUL_ENVIRONMENT=' ${PROJECT_DIR}/.env.local | cut -d= -f2-)
ENVIRONMENT=${ENVIRONMENT:-master}

status=$(curl -s -o /tmp/contentful-probe.json -w "%{http_code}" \
  -X POST "https://graphql.contentful.com/content/v1/spaces/${SPACE_ID}/environments/${ENVIRONMENT}" \
  -H "Authorization: Bearer ${DELIVERY_TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"query":"{ flexiblePageCollection(where: { slug: \"/\" }, limit: 1) { items { sys { id } slug pageTitle } } }"}')
echo "HTTP $status"
cat /tmp/contentful-probe.json
```

Classify the outcome of Check A:

- **`pass`** — 200 + non-empty `items`. Connectivity OK, credentials valid, sample entry present.
- **`pass-empty`** — 200 + empty `items`. Connectivity OK; sample entry missing (Step 6 was skipped or the entry was deleted).
- **`schema-missing`** — 200 + GraphQL error about `flexiblePageCollection` not existing. Content type wasn't created (Step 6 was skipped or failed).
- **`bad-token`** — 401. `CONTENTFUL_DELIVERY_TOKEN` is invalid.
- **`bad-space`** — 404. `CONTENTFUL_SPACE_ID` is invalid.
- **`other`** — anything else. Relay the response body verbatim.

**Check B — codebase compiles cleanly.** Always run.

```bash
npm --prefix ${PROJECT_DIR} run build
```

`next build` runs the TypeScript type-checker and ESLint. Any failure means the overlay didn't sit cleanly on top of the `create-next-app` scaffold (e.g. a Next.js API changed, the `@/*` alias didn't resolve, or a dependency is missing).

Classify the outcome of Check B:

- **`pass`** — exit 0.
- **`fail`** — non-zero. Surface the first error block. Do **not** attempt to auto-patch the scaffold; this is a signal that the skill template needs to be updated for the current `create-next-app@latest` output.

### Step 8 — Print next steps (do NOT run them)

Branch on the Check A / Check B results recorded in Step 7. Use the table below to choose the message. Print only the section that matches.

**Both green (`pass` + `pass`):**

```
✓ Validated against Contentful. ✓ Build clean.

Ready:
  cd <project-name>
  npm run dev
```

Then point the user at `components/ARCHITECTURE.md` for the six-step recipe to add their first section.

**Check A `pass-empty` or `schema-missing`, Check B `pass`:**

The app will run but `/` will return 404 because the home page entry doesn't exist (or the content types weren't created). Tell the user one of:
- Re-run Step 6 (`node ${PROJECT_DIR}/scripts/seed-contentful.mjs` with a CMA token), or
- Create the `flexiblePage` and `seo` content types and a `flexiblePage` entry with slug `/` manually in Contentful.

Then `npm run dev` will serve `/`.

**Check A `bad-token` or `bad-space`:**

Do **not** tell the user to run `npm run dev` — it will throw at request time. Tell them the specific credential to fix:
- `bad-token` → regenerate the Delivery token at Contentful → Settings → API keys, replace the value in `${PROJECT_DIR}/.env.local`, re-run validation.
- `bad-space` → fix `CONTENTFUL_SPACE_ID` in `${PROJECT_DIR}/.env.local`, re-run validation.

**Check A `not-run` (user skipped credentials in Gate 3):**

Tell the user they must fill in `CONTENTFUL_SPACE_ID` and `CONTENTFUL_DELIVERY_TOKEN` in `${PROJECT_DIR}/.env.local` before `npm run dev` will serve any page.

**Check B `fail` (regardless of Check A):**

Do **not** tell the user to run `npm run dev`. The build is broken; surface the error block and stop. The template needs a fix before this scaffold is usable.

## What this skill does NOT do

- Does not pin a Next.js version — relies on whatever `create-next-app@latest` ships.
- Does not ship example sections — only FlexiblePage + SEO queries. Sections are added per project per the architecture doc.
- Does not configure **section** content models. Step 6 creates `flexiblePage` and `seo` in the user's Contentful space when they provide a CMA token; any section types they build later (Hero, CardGroups, etc.) are their responsibility.
- Does not pull from any client-specific repo.
